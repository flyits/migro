最佳实践

遵循这些最佳实践，让你的数据库迁移更加安全和可维护。

1. 迁移命名

使用描述性名称，清晰表达迁移目的：

bash 
migro create create_users_table      # 创建表
migro create add_phone_to_users      # 添加列
migro create drop_legacy_columns     # 删除列
migro create add_index_to_posts      # 添加索引

2. 始终编写 Down 方法

确保每个迁移都可以回滚：

go 
func (m *Migration) Down(ctx context.Context, e *migrator.Executor) error {
    // 不要留空！
    return e.DropTableIfExists(ctx, "users")
}
⚠️ 重要

空的 Down 方法会导致无法回滚，在生产环境可能造成严重问题。

3. 使用事务

Migro 会自动为支持事务 DDL 的数据库（PostgreSQL/SQLite）使用事务，无需手动处理。

💡 提示

MySQL 的 DDL 语句会隐式提交事务，无法回滚。建议在执行前使用 --dry-run 预览。

4. 大表迁移

对于大表，避免锁表操作：

go 
// 使用 Raw SQL 执行在线 DDL（MySQL）
e.Raw(ctx, "ALTER TABLE users ADD COLUMN phone VARCHAR(20), ALGORITHM=INPLACE, LOCK=NONE")

// PostgreSQL 并发创建索引
e.Raw(ctx, "CREATE INDEX CONCURRENTLY idx_users_email ON users(email)")

5. 环境变量

生产环境使用环境变量存储敏感信息：

bash 
export DB_HOST=prod-db.example.com
export DB_PASS=secure_password
migro up

6. 预览 SQL

在生产环境执行前，先预览将要执行的 SQL：

bash 
migro up --dry-run

7. 备份数据库

在执行重要迁移前，始终备份数据库：

bash 
# MySQL
mysqldump -u root -p myapp > backup.sql

# PostgreSQL
pg_dump myapp > backup.sql

8. 保持迁移小而专注

每个迁移只做一件事：

  • ✅ 一个迁移创建一个表
  • ✅ 一个迁移添加一组相关的列
  • ❌ 避免在一个迁移中做太多事情
